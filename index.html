Storage & upsert behaviour

Every observation is an immutable Event = a platform-managed envelope wrapping a verbatim raw payload. There are no in-place updates in the lake. To change a row, you emit a new event with the same eventKey and a later occurredAt; the lake keeps both, and the serving layer resolves to the latest by primary key.

The connector sets eventKey, which is also the idempotency key — a duplicate inside the dedupe window is silently suppressed. At the serving boundary, write semantics are explicit: insert, upsertByPK (default), upsertByComposite, or rejectIfChanged for optimistic concurrency. Append-only + idempotent upsert is what makes replay, audit, and time-travel free.

append-onlyeventKey idempotencyupsertByPKtombstone deletesat-least-once + idempotent sink ⇒ exactly-once

C

Connector framework

Built-in connectors are Lattice-maintained and semver'd. Custom connectors are a TypeScript SDK in a sandboxed V8 isolate — Pullable, Webhookable, CDCStreamable, Introspectable — capability-scoped, packaged as a .connector bundle in a per-tenant registry.

S

Schema registry, advisory by design

Each type category carries an optional schema and two live views: the declared contract and the rolling discovered union of observed shapes. Evolution is additive and non-destructive — versions are tracked but never gate ingestion. One-click promote discovered → declared.

Architectural invariant The lake is the source of truth; the bus is an ephemeral broadcast. The DLQ catches envelope failures only — never schema mismatches — so a growing DLQ is a clean operational signal, not noise.
Layer 02Entities & Relationships

The ontology. Where the platform stops
being a lake and starts being a product.

Typed entities, first-class relationships, and materialized aggregate views — all keyed by a stable entityDefId, all queried through one ANSI SQL surface. This is the shared substrate everything above composes; it is never re-declared.

Two storage modes. No hybrid.

Read+write — the platform owns the data; app code, forms, workflows, and agents write through a typed API. Read-only — the platform stores it but a Transformation pipeline populates it from a source like Salesforce. The distinction is access-capability, not storage-origin — which keeps the model honest. From the query side, both are identical.

Four cardinalities, first-class.

1:1, 1:N, N:1, N:M — relationships are real objects with their own relDefId, auto-chosen storage (FK column or platform-managed junction), edge properties, and temporal validity (validFrom / validTo) from day one — not foreign-key strings hidden in JSON.

Derived, multi-entity datasets — joined, denormalized — that read like one entity but stay in sync with their sources. Declarative form or a controlled-SQL escape hatch. Incremental refresh in seconds.

Layer 04Transformation

A ladder, not a binary. From a one-line
filter to a managed Spark job.

Stream-to-entity pipelines hydrate the ontology from the lake. Entity-to-entity pipelines sync between entities (aggregate views compile to these). Both ride the same five rungs. The principle: people pick the lowest rung that works, and the platform makes the low rungs delightful so authors aren't tempted into code unless they truly need it.

ƒ

More complex — cloud functions

When declarative runs out, escape into a sandboxed TypeScript lambda per change, or an author-deployed managed Spark job for stateful, windowed, multi-stream work. Metered.

L1
Passthrough · free
Schemas already match — a 1:1 map, no logic. Auto-generated, one-click, no compute charge.
L2
Declarative map
Single source, per-field CEL derivations, type coercion, defaults, simple filters. Most apps' projection. Built-in engine.
L3
Multi-input join
Declarative joins, windowed aggregations, in-memory cached lookups (<100k-row reference entities). Same surface as the aggregate-view designer.
L4
Lambda per change · metered cloud function
Your TypeScript runs once per event in a V8 isolate — capability-scoped SDK lookups, returns 0/1/many rows. Stateless by design (cross-event state forces L5). Hard CPU & memory caps.
L5
Managed Spark · premium cloud function
Author-deployed Spark Structured Streaming (Scala / PySpark). Platform mounts sources as DataFrames, manages a tenant-isolated session, checkpoints, and auto-scales on backlog. You own the logic; the platform owns the cluster.
Correctness by construction Every sink upserts by primary key. Combine at-least-once consumption with an idempotent sink and you get effectively exactly-once everywhere — automatic for L1–L3, the lambda's return value for L4, a platform-provided idempotent writer for L5.

Backfill & rebuild are first-class

A stream-to-entity pipeline backfills from the Iceberg lake before switching to the live bus; an entity-to-entity pipeline backfills from source history. A rebuild runs into a shadow table and atomically swaps — the old version is retained for seven days. Bounded only by the upstream type's retention.

Layer 05Forms

Forms are tiny workflows.
Stop treating them like input widgets.

A form has fields — but also draft state, resume behaviour, per-step persistence, role-aware variants, and a commit moment that calls a domain operation. We treat all of that as first-class. The same model spans a single-page submit and a twelve-step wizard: single-page is just a one-step wizard.

Draft vs commit, per step

Draft steps write everything at submit, kept in a draft store keyed by formInstanceId. Commit steps write incrementally to the entity through its domain command, in a relaxed drafting state. A single wizard can mix both.

Revalidate on every resume

When a user returns to a draft, the platform re-runs validation against the current world and shows a “what changed” panel — the credit limit dropped, the state moved, the form version bumped. No silent stale assumptions. That's what makes drafts safe to keep for thirty days.

Definitions vs handles

Everything is keyed by a stable def ID — entityDefId, eventDefId, externalDefId. An SDK call takes a def ID and returns a typed handle bound to the request's tenant, subject, and access context — it cannot leak across requests. The schema registry generates per-tenant TypeScript types injected straight into every isolate.

tenant scopeaccess controlauditcost meteringOpenTelemetry

Every call passes that five-stage gate before it reaches a backing service. The SDK is the only way out of the isolate — no fetch(), no native driver. Outbound network is closed by default; an egress allowlist makes data exfiltration structurally impossible.

export default async (event, ctx) => {

  // db() — typed, tenant-bound, access-gated

  // external() — managed auth, invisible creds, allowlisted

                       .call({ amount: event.amount });

Layer 07Access Control

Three models, one engine.
There is no second authorization engine.

RBAC, ABAC, and ReBAC composed into a single decision engine that every layer consults with one question shape — (subject, action, resource, field?). Authors never write “RBAC” or “ABAC”; they write capability, conditions, rowAccess, and relationScope, and the engine maps each to a model. The default composition is AND — defense in depth.

R

RBAC — what you may attempt

Roles and permissions, stored as relationships. User —hasRole→ Role.

A

ABAC — under what conditions

CEL over subject, resource, and environment. The same CEL engine as form validation — learn it once.

Re

ReBAC — the right relationship

Traversal over Layer-02 relationships, accelerated by a derived access index. “Managers see their reports' deals.”

Row access compiles into the query

The post-fetch-filter anti-pattern is rejected outright — it yields short pages, lying counts, and leaky exports. Instead, row-level rules compile into the query before execution, against the same Nativa Query AST the entity layer and the SDK use. Field-level rules apply as a projection plus an obligation. Listing uses seek cursors that embed the policy version, so a mid-pagination policy change returns a clean 409 policy_changed rather than silently leaking.

predicate pushdownrelationship expansionaccess index JOINseek paginationreplayable decision log

appliesTo: { resource: Deal, action: read, mode: list }

  pushdown: true

Layer 08Custom Apps & Journeys · with Workflows

Apps are the shell.
Journeys are the work.

A Custom App is a branded, deployable surface — pages, role-aware navigation, search, theme. A Journey is a long-running, multi-actor, state-driven flow that lives inside it. Each state has a form chosen per role, commits a domain operation, and renders in the app. Three primitives — state × role × form — infinite shapes.

A journey is the experience around a workflow

The journey owns assignments, the inbox, per-state forms, the timeline, and closure. The Workflow Engine underneath owns durable execution — retries, compensation, signals, and timers that can wait days. Pairing them lets the experience iterate independently of the runtime.
